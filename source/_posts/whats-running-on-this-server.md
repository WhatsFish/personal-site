A quick map of every public-facing system on `20.89.176.30`, why each one
exists, and how they're wired together.

<!-- more -->

## The fleet

Five things share a single Azure VM behind one Nginx server block:

| URL | What it is | Stack |
|---|---|---|
| [/](http://20.89.176.30/) | This blog | Hexo (static) |
| [/traffic/](http://20.89.176.30/traffic/) | Web-server access-log report | GoAccess (static HTML, regenerated by cron) |
| [http://20.89.176.30:3000/](http://20.89.176.30:3000/) | Umami analytics dashboard | Docker container, direct port |
| [/chat](http://20.89.176.30/chat) | Personal AI chat | Next.js + Postgres, Azure Llama via Managed Identity |
| [/vpn](http://20.89.176.30/vpn) | Hysteria2 VPN traffic monitor | Next.js + Postgres + a host-side collector |

Everything except Umami is exposed by path under the same hostname, so the
public surface stays small (TCP/80 + UDP/443 for the VPN itself, plus TCP/3000
for Umami).

## How routing works

There's a single Nginx server block for port 80:

```nginx
server {
    listen 80 default_server;
    server_name _;
    include snippets/traffic-monitor.conf;   # /traffic/...
    include snippets/ai-playground.conf;     # /chat
    include snippets/vpn-monitor.conf;       # /vpn

    root /home/liharr/src/personal-site/public;
    location / { try_files $uri $uri/ =404; }
}
```

Each `snippets/*.conf` is a **symlink into the matching git repo**, so the
routing config travels with the application code. Adding a new portal is a
five-step ritual:

1. Write `nginx/<name>.conf` in the new repo (`location /<name> { proxy_pass ... }`).
2. Bind the container to a free `127.0.0.1:<port>`.
3. Symlink it into `/etc/nginx/snippets/`.
4. Add one `include` line to the personal-site server block.
5. `sudo nginx -t && sudo systemctl reload nginx`.

## Each piece, briefly

### Blog (Hexo)

Plain static files. Nginx serves `~/src/personal-site/public/`. Builds happen
locally with `hexo generate`. The repo lives at
[github.com/WhatsFish/personal-site](https://github.com/WhatsFish/personal-site).

### Umami analytics

A self-hosted, privacy-friendly Google-Analytics-alternative. The container
ships with its base path baked in at build time, so I gave up on reverse-proxying
it and just exposed port 3000 directly. The blog and `/chat` both inject a
small `<script>` tag pointing at `:3000/script.js`.

Two separate Umami "websites" so I can split traffic by app:

- Blog: `3816fe35-...`
- ai-playground: `5b77c34e-...`

### GoAccess report

Server-side log analyzer. A cron job runs every 5 minutes, reads
`/var/log/nginx/personal-site.access.log`, and writes a static HTML report
into a directory Nginx aliases at `/traffic/`. Zero JS, zero database, just
fast facts about what hit the box.

### ai-playground (`/chat`)

A small Next.js 14 app I built to talk to Azure-hosted models without putting
an API key in plaintext. Auth is GitHub OAuth, gated by an allowlist. The
auth route handler re-prepends the `/chat` basePath to inbound URLs because
Next.js strips it before invoking handlers — Auth.js then matches its
configured basePath at both ends (action parsing *and* OAuth callback URL).

The Azure side is the more interesting bit. Instead of long-lived API keys,
the VM has a system-assigned **managed identity** with a `Cognitive Services
User` role on the AI resource. The app calls `DefaultAzureCredential` →
hits the IMDS endpoint → gets a short-lived bearer token → calls the
`https://<resource>/openai/v1/chat/completions` endpoint with `stream: true`.
Token refresh is cached in-process. No secrets in env files, no rotation.

### vpn-monitor (`/vpn`)

A dashboard for the Hysteria2 VPN that already runs on the same host. Two
moving parts:

1. **Collector** — a tiny systemd service on the host. Polls Hysteria's
   `trafficStats` HTTP API every 30s for cumulative byte counts and active
   connection counts, and tails `journalctl -fu hysteria-server -o cat` for
   `client connected` / `client disconnected` / `authentication failed` events.
   Source IPs go through `ip-api.com` for country/region/ISP data, with a
   Postgres cache to stay under the free tier's rate limit.
2. **Web** — same Next.js + Auth.js skeleton as ai-playground, just pointed at
   the collector's tables. Two pages: an Overview (cumulative tx/rx, current
   throughput, 24h connect/disconnect/auth-failed counts) and a Connections
   table.

The collector lives on the host (not in a container) because reading
journald from inside Docker is annoying — you need the right machine-id, the
journal directory mounted, and `journalctl` available. A 30-line systemd
unit is just simpler.

## Shared Postgres

All three database-backed services (Umami, ai-playground, vpn-monitor) talk
to the **same Postgres container** spun up by the `traffic-monitor` repo. They
each get their own database and role. The container is bound to
`127.0.0.1:5432` so host-side services (the vpn-monitor collector) can reach
it, but it's invisible from the public internet.

That's a deliberate trade-off: one shared Postgres means slightly more
coupling between repos, but it saves a chunk of RAM versus running three
separate database containers on a small VM.

## Port allocation

| Port | Where |
|---|---|
| 80 / TCP | Nginx public entrypoint |
| 443 / UDP | Hysteria2 VPN |
| 3000 / TCP | Umami (public) |
| 3001 / TCP | ai-playground web (loopback) |
| 3002 / TCP | vpn-monitor web (loopback) |
| 5432 / TCP | Postgres (loopback) |
| 7653 / TCP | Hysteria trafficStats (loopback) |

## What I'd add next

- **HTTPS**. Currently everything is plain HTTP. Putting Caddy or
  nginx + Let's Encrypt in front of port 80 is the obvious next step.
  Once that lands, Umami's mixed-content `:3000` script include needs to
  move behind the same TLS layer.
- **A landing page at `/`** that links to all the portals (blog post about
  this whole setup, basically the table at the top).
- **vpn-monitor v2**: per-user accounting (requires migrating Hysteria from
  single-password to multi-user auth), historical throughput charts, and a
  real map for the Connections page.
